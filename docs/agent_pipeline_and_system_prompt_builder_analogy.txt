================================================================================
PIPELINE АГЕНТІВ: EXAI_Agents (Streamlit) vs system_prompt_builder.py
================================================================================
Оновлено під фактичну оркестрацію в app.py + state_machine.py + agents/runner.py.
system_prompt_builder.py у репозиторії не імпортується з app.py — це паралельний шар
промптів для іншого рушія (7 стадій); нижче — зіставлення за змістом і відмінності.

--------------------------------------------------------------------------------
1. АЛГОРИТМ EXAI_Agents (актуальний)
--------------------------------------------------------------------------------

Джерела
  • Корпус тем: content/research_blocks.json — блок, phases[] з phase_id та title.
    Текстів готових питань у JSON немає; питання генерує LLM (CANONICAL_Q) з опорою
    на block title, phase title, крок general|deepening|drilling, фокус з A16 та історію.
  • Промпти: agents/prompts.py (SYSTEM_*).
  • Виклики: agents/runner.py → OpenRouter; мова — response_language + суфікс у runner.
  • Оціночний поріг скопінгу: score_threshold (state_machine, сайдбар); floor розуміння
    відповіді: UNDERSTANDING_FLOOR = 0.7 для гілки A13.
  • Тривалість сесії в тексті: content/research_loader.py — DEFAULT_SESSION_MINUTES (фіксовано).

Модель «стадій» інтерв’ю
  • Одна стадія сесії = одна phase з JSON; послідовність фаз — за block.phases.
  • У кожній фазі рівно 3 кроки: general → deepening → drilling (див. state_machine.STEP_ORDER).
  • У поточному app після скопінгу завжди обираються усі фази блоку (all_phase_indices).

Схема потоку (ASCII)

  [SETUP] користувач: блок, модель, мова
       |
       v
  A14 (preset topic)  --->  USER (роль / досвід)
       |
       v
  A16 (JSON, не в чаті) ---> decide_after_a16
       |
       +-- answer_understanding < 0.7 ----> A13 ---> USER (цикл знову через A16)
       |
       +-- should_agent_reask або слабкий purpose/focus vs threshold ----> A17 ---> USER (цикл)
       |
       +-- OK ----> A18 (огляд фаз, без питань на зміну плану)
                 |
                 +----> у ТОМУ Ж ході: main_phase = CANONICAL
                       + перше питання: CANONICAL_Q (_display_canonical_question)

  [CANONICAL] для кожного (phase, step):
       показати питання (заголовок фази + мітка кроку + текст від CANONICAL_Q)
       USER
       CANONICAL_DEPTH (JSON) — завжди, включно з відповіддю після follow-up (той самий canonical question у qtext)
            +-- should_reask=1 і deep<0.7 і canonical_follow_ups_used < 3: follow_up в чат; ++canonical_follow_ups_used
            +-- інакше: advance_canonical_position; awaiting_canonical_reask=false; canonical_follow_ups_used=0
       між фазами (next_phase): A22 (міст) + наступне питання CANONICAL_Q
       після останнього кроку: A11

Примітки
  • A22 — лише міст між фазами (run_a22); питання одразу мовою сесії з CANONICAL_Q.

--------------------------------------------------------------------------------
2. АЛГОРИТМ system_prompt_builder.py (структура)
--------------------------------------------------------------------------------

Функція: prompt_builder(stageNumber, stageOutcomes, language)

Що повертається
  • Для stageNumber 1..7 — словник stages[str(N)] (підмножина ключів a_*_prompt для цієї стадії),
    у який злито stage_final_step:
      a_21_prompt — одне «глибоке» інтерв’ю-питання на крок (контекст поточної стадії).
      a_22_prompt — JSON-оцінка глибини відповіді (should_agent_reask, deep_knowledge_level, …).
      a_23_prompt — текстовий follow-up за low_score_reason (уточнення).
    Додаткові ключі залежать від стадії (наприклад a_24 наприкінці Stage 1 — вхід у Stage 2).
  • outcomesDescription пояснює накопичуване поле outcomes (метрики з різних стадій).
  • languagePrompt — жорстка мова відповіді.

Хто крутить граф
  • Зовнішній застосунок сам вирішує, який a_* викликати після відповіді юзера і як
    оновлювати outcomes; builder лише видає тексти промптів на кожну з 7 стадій.

Стадії (фіксовані назви в builder)
  Stage 1 Introduction & Scoping … Stage 7 Conclusion — не прив’язані до research_blocks.json
  у цьому репозиторії.

--------------------------------------------------------------------------------
3. ТРИ ТАБЛИЦІ: ЛИШЕ PIPELINE АГЕНТІВ / ПРОМПТІВ
--------------------------------------------------------------------------------
Нижче — послідовність без змішування двох систем в одній сітці. Порівняння ролей
див. розділ 4.

Таблиця 1. EXAI_Agents (app.py) — хто за чим іде
───────────────────────────────────────────────

Частина А. Скопінг (до фази CANONICAL)

  №   Агент          У чат?   Що далі (типово)
  ──  ─────────────  ───────  ─────────────────────────────────────────────
  1   A14            так      очікується відповідь USER
  2   A16            ні       гілка: A13 або A17 або рядок 3
  —   A13            так*     USER → знову рядок 2 (*якщо незрозуміло текст)
  —   A17            так*     USER → знову рядок 2 (*якщо слабкий фокус / re-ask)
  3   A18            так      у тому ж обробнику одразу CANONICAL_Q (рядок Б1)

Частина Б. Canonical (кожна фаза JSON: general → deepening → drilling)

  №   Агент           У чат?   Що далі (типово)
  ──  ──────────────  ───────  ─────────────────────────────────────────────
  B1  CANONICAL_Q     так      текст питання → USER
  B2  CANONICAL_DEPTH ні       JSON; або follow-up у чат, або перехід далі
  —   (текст з        так*     один re-ask на крок (*умова в app.py)
       CANONICAL_DEPTH)
  B3  USER (відповідь після re-ask)  —    advance без повторного B2
  B4  A22             так*     міст між фазами (*лише при переході next_phase)
  B5  A11             так      кінець сесії (усі кроки пройдено)


Таблиця 2. system_prompt_builder — Stage 1 (ключі в stages['1'], порядок у файлі)
────────────────────────────────────────────────────────────────────────────────
Один з двох перших ключів на вибір рушія. Порядок рядків = порядок ключів у словнику;
фактичні гілки (наприклад після a_16: спочатку a_17 чи одразу a_18) визначає зовнішній
рушій, а не цей список.

  №   Ключ           Вихід LLM   Коротка роль
  ──  ─────────────  ──────────  ─────────────────────────────────────────────
  1a  a_14_prompt    текст       відкриття з пресетом main_topic
  1b  a_15_prompt    текст       відкриття без пресету теми
  2   a_16_prompt    JSON        аналіз відповіді на opening
  3   a_17_prompt    текст       follow-up по фокусу (після слабких скорів)
  4   a_18_prompt    текст       перехід: готовність обговорити фокус глибше
  5   a_19_prompt    JSON        аналіз відповіді після a_18 (scope / узгодження)
  6   a_20_prompt    текст       follow-up, якщо a_19 вказав на неясність
  —   (далі злито stage_final_step — ті самі a_21…a_23, що й у табл. 3)
  7   a_21_prompt    текст       одне інтерв’ю-питання (final step)
  8   a_22_prompt    JSON        глибина відповіді, re-ask
  9   a_23_prompt    текст       текстовий follow-up за low_score_reason
  10  a_24_prompt    текст       перехід до Stage 2 (наступна стадія продукту)

  Примітка: окремого ключа a_13 у builder немає; «уточнити незрозуміле» рушій
  може реалізувати іншим промптом або повторним викликом a_17 тощо.


Таблиця 3. system_prompt_builder — цикл питання всередині стадії (stage_final_step)
──────────────────────────────────────────────────────────────────────────────────
Цей блок зливається в stages['1']…['6'] (і подібно в інших стадіях). Рушій крутить
цикл, доки не виконані цілі стадії; потім — ключі на кшталт a_24 (залежно від N).

  Крок   Ключ          Вихід LLM   Хто відповідає / логіка
  ─────  ────────────  ──────────  ───────────────────────────────────────────
  C1     a_21_prompt   текст       USER (одне питання експерту)
  C2     a_22_prompt   JSON        рушій оцінює відповідь
  C3     a_23_prompt   текст*      USER* (*якщо потрібен follow-up за a_22)
  C4     (повтор)      —           знову C1 з оновленою історією / outcomes

  Після завершення стадії зовнішній код перемикає stageNumber (наприклад Stage 1
  → a_24_prompt, далі Stage 2 зі своїм набором a_16…).

--------------------------------------------------------------------------------
4. ТАБЛИЦЯ: ВІДПОВІДНІСТЬ ІМЕН (з обмеженнями)
--------------------------------------------------------------------------------

EXAI (runner / лог)     | Зв’язок із system_prompt_builder
------------------------|--------------------------------------------------
A14                     | a_14_prompt (Stage 1) — відкриття; a_15_prompt у builder залишається для інших рушіїв, у EXAI не викликається
A16                     | a_16_prompt (Stage 1) — JSON-аналіз opening; схема полів інша
A13                     | Логіка уточнення змістово як навколо a_16
A17                     | a_17_prompt — follow-up по фокусу
A18                     | НЕ еквівалент Stage 1 a_18: у EXAI — roadmap фаз JSON; у builder a_18 — перехід до глибини фокусу
A22                     | У EXAI — міст між фазами корпусу. У builder a_22 з stage_final_step — JSON глибини (аналог CANONICAL_DEPTH)
CANONICAL_Q             | Немає окремого ключа; найближче за роллю — динамічне питання з a_21 + корпусна прив’язка EXAI
CANONICAL_DEPTH         | Змістово як a_22 з stage_final_step (JSON + re-ask)
A11                     | Тон і роль закриття як у Stage 7 / conclusion-промптах
