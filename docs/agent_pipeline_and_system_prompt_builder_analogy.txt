================================================================================
PIPELINE АГЕНТІВ (Streamlit-додаток EXAI_Agents) ТА АНАЛОГІЯ З system_prompt_builder.py
================================================================================
Документ описує послідовність роботи агентів у поточному проєкті та показує, як
вона відповідає (і чим відрізняється від) логіки з файлу system_prompt_builder.py.

--------------------------------------------------------------------------------
1. ЩО РОБИТЬ ПОТОЧНИЙ ДОДАТОК (app.py + state_machine + agents/runner)
--------------------------------------------------------------------------------

1.1 Джерела даних
- Канонічні питання інтерв’ю: content/research_blocks.json (блок → фази → кроки
  general / deepening / drilling). Модель їх НЕ вигадує — текст показується з JSON.
- Системні промпти агентів: agents/prompts.py (константи SYSTEM_A14 … SYSTEM_A22,
  SYSTEM_CANONICAL_DEPTH).
- Виклик LLM: agents/runner.py через OpenRouter (OpenAI-сумісний клієнт).

1.2 Фази життєвого циклу сесії (високорівнево)

  [Налаштування]
    Користувач обирає модель OpenRouter, блок дослідження (1–7), чи використовувати
    пресет теми (A14) чи узгодження теми (A15).

  [Фаза SCOPING — діаграма A14–A22]
    Step 1 (мета та фокус):
      - Після старту: A14 або A15 → перше повідомлення асистента.
      - Відповідь користувача → A16 (JSON: оцінки, extracted_focus_area, …).
      - Гілки за state_machine.decide_after_a16:
          * answer_understanding_score < 0.7 → A13 (уточнення)
          * purpose та focus нижче порогу з сайдбару → A17 (follow-up)
          * інакше → Step 2, A18 (пропозиція scope по фазах блоку)

    Step 2 (узгодження scope сесії):
      - Відповідь на A18 → A19 (JSON: scope_agreement_score, scope_areas, …).
      - Гілки за decide_after_a19:
          * незрозуміло → A13
          * scope_agreement низький → A20
          * інакше → Step 3, A21 (перелік канонічних питань verbatim + підтвердження)

    Після A21:
      - Кнопки: продовжити до канонічних питань або завершити (A11).

  [Фаза CANONICAL]
    Для кожної обраної фази та кожного кроку (general → deepening → drilling):
      - Показується текст з research_blocks.json.
      - Відповідь → оцінка глибини (аналог a_22): SYSTEM_CANONICAL_DEPTH у runner.
      - За потреби один re-ask (follow_up_question), далі перехід до наступного кроку.
    Між фазами: A22 (короткий міст).
    Після останнього кроку: A11 (фінальне повідомлення).

1.3 Відповідність імен: EXAI_агент ↔ ключ у system_prompt_builder (за змістом)

  EXAI (код)          system_prompt_builder.py (типові ключі)
  ----------------    --------------------------------------------------
  A14 / run_a14       a_14_prompt (Stage 1) — перше питання / відкриття
  A15 / run_a15       a_15_prompt — відкриття без пресету теми
  A16 / run_a16       a_16_prompt — аналіз відповіді (JSON, скори, фокус)
  A13 / run_a13       a_13_prompt (у великому файлі для різних гілок) —
                      «чи я правильно зрозумів» / уточнення
  A17 / run_a17       a_17_prompt — follow-up після відкриття
  A18 / run_a18       логічно близько до кроків пропозиції scope (у builder є
                      різні a_* для уточнення фокусу/скоупу в межах стадії 1)
  A19 / run_a19       a_19_prompt — аналіз відповіді на пропозицію (JSON)
  A20 / run_a20       a_20_prompt — текстовий follow-up до уточнення фокусу/скоупу
  A21 / run_a21       у builder немає прямого «перелік підпунктів» з зовнішнього
                      JSON; натомість далі йдуть кроки стадії. У EXAI A21 явно
                      виводить список питань з JSON.
  A11 / run_a11       a_11_prompt / a_18_prompt (залежно від стадії) — прощання
  A22 / run_a22       перехід до наступної фази; у builder є схожі «мости» між
                      кроками в межах 7 стадій
  Depth judge         a_22_prompt у stage_final_step — глибина + should_agent_reask
                      + JSON; у EXAI спрощено до SYSTEM_CANONICAL_DEPTH

У поточному додатку номери «A14» тощо — це зручні мітки в коді; у builder ключі
мають вигляд a_14_prompt і повертаються з prompt_builder() у словнику.


--------------------------------------------------------------------------------
2. ЯК ПРАЦЮЄ system_prompt_builder.py (структура)
--------------------------------------------------------------------------------

2.1 Функція
  prompt_builder(stageNumber, stageOutcomes, language)

2.2 Що вона будує
- Мовний суфікс: languagePrompt (жорстка мова відповіді).
- outcomesDescription: великий текст про поле outcomes — які метрики накопичуються
  після стадій 1–6; розширюється залежно від stageNumber (additionalOutcomes1…5).
- stage_final_step — словник промптів, який підмішується в КОЖНУ стадію:
    a_21_prompt — одне «глибоке» інтерв’ю-питання на крок (динамічно під поточну
                  стадію через currentStageText[stageNumber-1])
    a_22_prompt — аналітик: JSON (deep_knowledge_level, should_agent_reask, …)
    a_23_prompt — текстовий follow-up за low_score_reason (аналог «допитати»)

- stages — словник '1'…'7': для кожної стадії свій набір a_XX_prompt (a_14, a_15,
  a_16, … a_20, a_24, тощо залежно від стадії).

2.3 Повернення
  prompts = stages.get(str(stageNumber), {})
  return prompts

Тобто для стадії N повертається лише підсловник цієї стадії (у який уже злито
**stage_final_step, тобто a_21/a_22/a_23 присутні разом з a_14…).

2.4 Хто викликає prompt_builder у продакшені
У цьому репозиторії app.py НЕ імпортує system_prompt_builder.py — це окремий/наслідковий
модуль (імпорт з ...agents вказує на пакетну структуру іншого проєкту). Аналогія
змістовна: обидва підходи описують «набір системних промптів на крок інтерв’ю».


--------------------------------------------------------------------------------
3. ПОРІВНЯЛЬНА ТАБЛИЦЯ: ОДНА І ТА САМА ІДЕЯ, РІЗНА РЕАЛІЗАЦІЯ
--------------------------------------------------------------------------------

Аспект                          EXAI Streamlit app              system_prompt_builder
------------------------------  -----------------------------  ---------------------------
Оркестрація                     Python: app.py + state_machine Зовнішній код викликає
                                (явний граф станів)            prompt_builder(stage,…)
Канонічний контент              research_blocks.json           Зазвичай всередині промптів
                                (блок/фаза/крок)               або логіка стадій 1–7
Кількість «рівнів»              Блок → фази → 3 кроки          7 стадій × багато підкроків
Мова UI / промптів              Зараз English у prompts.py   language у prompt_builder
Фінальні кроки стадії           Depth + optional re-ask        a_21 + a_22 + a_23
Поріг «достатньо добре»         score_threshold (сайдбар)      Зашиті правила в тексті a_22
Вихід аналізатора               JSON у A16, A19, depth         JSON у a_16, a_19, a_22 тощо


--------------------------------------------------------------------------------
4. ПОСЛІДОВНІСТЬ (PIPELINE) У ВИГЛЯДІ СХЕМИ (EXAI)
--------------------------------------------------------------------------------

  START
    → [A14 | A15] відкриття
    → USER
    → [A16] оцінка
         → [A13] якщо незрозуміло
         → [A17] якщо слабкий purpose/focus
         → [A18] якщо OK
    → USER
    → [A19] оцінка scope
         → [A13]
         → [A20]
         → [A21] список питань з JSON
    → ВИБІР: продовжити | A11
    → CANONICAL: питання з JSON
         → [depth] → можливий re-ask
         → [A22] між фазами
    → [A11] кінець


У system_prompt_builder відповідна логіка розмазана по стадіях 1–7 і по ключах
a_*; зовнішній рушій повинен вирішувати, який ключ викликати після відповіді юзера
та як оновлювати outcomes — у EXAI те саме роль виконують FlowState + функції
_handle_scoping_user_message / _handle_canonical_user_message.


--------------------------------------------------------------------------------
5. ФАЙЛИ ДЛЯ ПОДАЛЬШОГО ЧИТАННЯ
--------------------------------------------------------------------------------

- app.py                    — UI, session_state, виклики runner, пороги
- state_machine.py          — decide_after_a16 / decide_after_a19, навігація по фазах
- agents/prompts.py         — текст SYSTEM_* (аналог тіл a_*_prompt)
- agents/runner.py          — збір user-повідомлення + виклик LLM
- content/research_blocks.json — канонічні питання (у builder такого окремого шару немає)
- system_prompt_builder.py  — монолітний генератор усіх a_*_prompt для стадій 1–7
